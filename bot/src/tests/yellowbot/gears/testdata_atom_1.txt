<h2>Erik Montnemery joins Nabu Casa</h2>
<p>Today, we are proud to announce that Erik Montnemery has joined Nabu Casa
to work full-time on the Home Assistant project.</p>
<p>Some of you might know Erik better by his GitHub handle: <a href="https://github.com/emontnemery">@emontnemery</a>, or
by one of the integrations he works on: <a href="/integrations/cast">Cast</a>,
<a href="/integrations/tasmota">Tasmota</a> and <a href="/integrations/mqtt">MQTT</a>!</p>
<p>We are excited to have you aboard Erik! Welcome!</p>
<h2>Python 3.7 dropped, Python 3.9 supported</h2>
<p>Support for Python 3.7 has been removed, raising the minimum Python version
you need to run Home Assistant Core to Python 3.8. Python 3.7 was already
deprecated since Home Assistant Core 0.116.0.</p>
<p>This is only a concern if you are running Home Assistant Core directly on
Python. If you run Home Assistant OS, Supervised or Container, you don‚Äôt have
to worry about this, as the Home Assistant project takes care of this for you.</p>
<p>However, say hello to: Python 3.9. As of this release, Python 3.9 is fully
supported and tested.</p>
<h2>Z-Wave JS</h2>
<p>This release brings you the <a href="/integrations/zwave_js">Z-Wave JS integration</a>!</p>
<p>A brand new integration for Z-Wave in Home Assistant, full-blown, created
in just a month by a bunch of awesome people!</p>
<p>More and more people were concerned about the future of Z-Wave with
Home Assistant; meanwhile the <a href="https://zwave-js.github.io/node-zwave-js/#/">Z-Wave JS</a> project was rapidly growing
and gathering a large community around it. Long story short: Home Assistant and
Z-Wave JS teamed up! And a lot of contributors jumped on the train!</p>
<p>This new integration is based on the same base principles as the OpenZWave
integration: It is decoupled from Home Assistant. Instead of MQTT, the Z-Wave
JS integration uses a WebSocket connection to a Z-Wave JS server.</p>
<p>This means, in order to use this new integration, you‚Äôll need to run the
Z-Wave JS server that sits in between your Z-Wave USB stick and Home Assistant.
There are multiple options available for running the Z-Wave JS server,
via Docker or manually, and there is also a Home Assistant add-on available.</p>
<p>If you run an installation with a Supervisor, the add-on will even be
set up automatically for you.</p>
<h3>Current state of Z-Wave JS</h3>
<p>The new Z-Wave JS integration is amazingly complete. It supports the following
platforms from the start: Binary Sensors, Climate, Covers, Fans, Lights, Locks,
Sensors and Switches. Scenes are also supported!</p>
<p>Besides that, it is just <em>blazing</em> fast! ‚ö°Ô∏è</p>
<p>While fast-growing, it is a new integration and there is still a lot
in progress. Of course, no doubt, there will be bugs. Bugs that will
be squashed and handled. Please, be sure to
<a href="https://github.com/home-assistant/core/issues/new/choose">report issues in our issue tracker</a>.</p>
<p>What is left? See our <a href="https://github.com/home-assistant-libs/zwave-js-server-python/issues/56">Z-Wave JS integration roadmap</a>
for those details.</p>

<h3>What about the old Z-Wave integration?</h3>
<p>The old <a href="/integrations/zwave"><code>zwave</code></a> integration has been based on
OpenZWave 1.4, which is really old and does not get any upstream updates
anymore. This also means that it is hard for us to keep it in a working state
for the upcoming future.</p>
<p>Therefore, as of today, the old Z-Wave is marked as being <strong>deprecated</strong>.</p>
<p>If one is starting with Home Assistant and Z-Wave, the <code>zwave</code> integration is
no longer recommended. For existing users, we recommend migrating to the
<a href="/integrations/zwave_js">Z-Wave JS integration</a>. However, don‚Äôt panic,
<code>zwave</code> is deprecated but not removed yet. We currently have no plans for
removing it, unless it can no longer be made to be build for the
latest Python</p>
<h3>What about the OpenZWave beta integration?</h3>
<p>Over a year ago, Home Assistant started working on the
<a href="/integrations/ozw">OpenZWave integration</a>. Based on OpenZWave 1.6 and the
OZW daemon, using MQTT as a transport. This project was promising and something
we put a lot of time and effort in.</p>
<p>Unfortunately, the OpenZWave project itself became a bit stale as most
of the upstream development is done by a single person:
Justin Hammond (Fishwaldo).</p>
<p>Quite a few people were worried about the lesser activity on OpenZWave.
Justin <a href="https://community.home-assistant.io/t/can-we-revisit-the-move-to-qt-openzwave/262612/140">has responded on our forum</a> on what is happening in his life
and it is sad to read how many people have personally impacted him negatively.</p>
<blockquote>
<p>Is OZW Dead - No. I‚Äôll tinker with it, I‚Äôll make changes as time allows,
but it will be at my own pace, to scratch my own itches now.</p>
</blockquote>
<p>Let us be straight here: OpenZWave is an 11 year-old project, which is used
by an incredible amount of systems (besides Home Assistant) in this world,
with an even greater track record. So:</p>
<p><strong>Thank you Justin for all you have done in all those years!</strong></p>
<p>The team that was working on the OZW integration have shifted their focus to
the new Z-Wave JS integration. We don‚Äôt think the OpenZWave integration will
make it out of beta. We recommend Home Assistant users migrate to the
Z-Wave JS integration.</p>
<h2>No more badges in auto generated dashboards</h2>
<p>If you are using an auto generated Lovelace dashboard, you probably always had
a ton of badges on the top of your dashboard. If there are just a few it works,
but as more start to appear, it soon becomes cluttered.</p>
<p class='img'>
<img src='/images/blog/2021-02/badges.png' alt='Screenshot of the old badges clutter'>
Screenshot of the old badges clutter that is now gone!
</p>
<p>We decided to no longer use badges in the auto generated Lovelace dashboards.
Instead, every entity is neatly grouped by area or domain in an entities card.</p>
<p>If you liked the badges, you can always take control of your dashboard
and add them.</p>
<h2>DHCP Discovery</h2>
<p>Home Assistant now listens on your network for devices that request a
IP address. Based on those intercepted messages, Home Assistant can now
discover integrations for you to use with the discovered devices.</p>
<p>This really powerful addition has been made by <a href="https://github.com/bdraco">@bdraco</a>, and many
integrations have quickly enriched their integrations to use this new
discovery method.</p>
<p>So as of this release, your Home Assistant might actually find some new
devices it can use! Thanks <a href="https://github.com/bdraco">@bdraco</a>!</p>

<p>Nabu Casa has been offering an amazing text to speech service for a while now,

panel.</p>

<h2>Breaking Changes</h2>
<p>Below is a listing of the breaking change for this release, per subject or
integration. Click on one of those to read more about the breaking change
for that specific item.</p>
<details>
  <summary><b>Python 3.7</b></summary>
  <p>
<p>Support for Python 3.7 has been dropped, after being deprecated since
Home Assistant 0.116.</p>
<p>This is only a concern if you are running Home Assistant Core directly on
Python. You can check the Python version Home Assistant uses by going to the
information screen in Configuration -&gt; Info.</p>
<p>If you run Home Assistant OS, Supervised or Container, you don‚Äôt have to worry
about this, as the Home Assistant project takes care of this for you.
As a matter of fact, if you use one of those methods, you are already
running Python 3.8.</p>
<p>(<a href="https://github.com/frenck">@frenck</a> - <a href="https://github.com/home-assistant/core/pull/43805">#43805</a>)</p>
  </p>
</details>
<details>
  <summary><b>Z-Wave</b></summary>
  <p>
<p>The old <code>zwave</code> integration is now considered legacy and deprecated. This
integration is still based on OpenZwave 1.4, which has been out of active
maintenance for quite some time now.</p>
<p>Alternatively, you can move to one of the other Z-Wave solutions Home Assistant
offers:</p>
<ul>
<li>
<p>OpenZWave (currently not maintained)</p>
</li>
<li>
<p>Z-Wave JS (introduced in this release, recommended)</p>
</p>
</li>
</ul>
</details>
<details>
  <summary><b>PTVSD</b></summary>
  <p>
<p>The PTVSD extension has deprecated a couple of releases back and is now removed.
The debugpy integration is a full replacement for this integration.</p>
<p>(<a href="https://github.com/frenck">@frenck</a> - <a href="https://github.com/home-assistant/core/pull/44748">#44748</a>)</p>
  </p>

  <summary><b>Tado</b></summary>
  <p>
<p>Tado‚Äôs primary functionality has fully transitioned to configuration via UI.
YAML configuration has been removed except for <code>device_tracker</code>. Existing YAML
configuration has already been imported automatically in the previous releases
and can now safely be removed from your configuration files.</p>
<p>The <code>device_tracker</code> platform continues to use YAML, and will be updated at a
later date when the device tracker platform is modernized.</p>
<p>Tado zone sensors have been replaced by their corresponding <code>binary_sensors</code>.
Therefore, you‚Äôll have to update Lovelace panels, scripts and automations
referring to the following sensors by their corresponding binary sensor

<li><code>sensor.ZONE_open_window</code> is replaced by <code>binary_sensor.ZONE_open_window</code></li>
<li><code>sensor.ZONE_overlay</code> is replaced by <code>binary_sensor.ZONE_overlay</code></li>
<li><code>sensor.ZONE_power</code> is replaced by <code>binary_sensor.ZONE_power</code></li>
</ul>
<p>The following state attributes have been removed:</p>
<ul>
<li><code>sensor.ZONE_overlay</code>: termination</li>
<li><code>sensor.ZONE_open_window</code>: open_window_attr</li>
</ul>
<p>(<a href="https://github.com/Noltari">@Noltari</a> - <a href="https://github.com/home-assistant/core/pull/44576">#44576</a>) (<a href="https://github.com/bdraco">@bdraco</a> - <a href="https://github.com/home-assistant/core/pull/45384">#45384</a>) (<a href="/integrations/tado/">tado docs</a>)</p>

<p>(<a href="https://github.com/bdraco">@bdraco</a> - <a href="https://github.com/home-assistant/core/pull/45381">#45381</a>) (<a href="/integrations/powerwall/">powerwall docs</a>)</p>
  </p>
</details>
<details>
  <summary><b>Light Profiles</b></summary>
  <p>
<p>Nominally a breaking change. A default profile for the light, now is always
applied, so controlling a light from UI would set all the parameters defined
in the profile.</p>
<p>For example, if the default profile for a particular light is set to:</p>
<ul>
<li>Color: Green</li>
<li>Brightness: 255</li>
<li>Transition: 5</li>
</ul>
<p>And the current light status is: Color - red, brightness - 20 and in UI you
change the brightness to 100 with the intention to only change the brightness,
then the default profile would also apply the color and transition.</p>
<p>(<a href="https://github.com/Adminiuga">@Adminiuga</a> - <a href="https://github.com/home-assistant/core/pull/45450">#45450</a>) (<a href="/integrations/light/">light docs</a>)</p>
  </p>
</details>
<details>
  <summary><b>MQTT</b></summary>
  <p>
<p>MQTT allowed passing an arbitrary speed that was not in the <code>speed_list</code>.</p>
<p>With the conversion to percentages in upcoming releases, this aberration from
the spec could not be converted to a percentage. Since it does not
conform to the entity model, it is not possible to preserve this exception.</p>
<p>(<a href="https://github.com/bdraco">@bdraco</a> - <a href="https://github.com/home-assistant/core/pull/45445">#45445</a>) (<a href="/integrations/mqtt/">mqtt docs</a>)</p>
  </p>
</details>
<details>
  <summary><b>NuHeat</b></summary>
  <p>
<p>NuHeat has fully transitioned to configuration via UI. YAML configuration has
been removed. Existing YAML configuration has already been imported
automatically in the previous releases and can now safely be removed from your
configuration files.</p>
<p>(<a href="https://github.com/bdraco">@bdraco</a> - <a href="https://github.com/home-assistant/core/pull/45380">#45380</a>) (<a href="/integrations/nuheat/">nuheat docs</a>)</p>
  </p>
</details>
<details>
  <summary><b>Rheem EcoNET Water Products</b></summary>
  <p>
<p>EcoNET has swirtched over to EcoNET‚Äôs new API.</p>
<p>Water Heaters will no longer report current tank
temperature. The add/remove vacation services are also removed,
alternatively, you can use away mode now.</p>
<p>(<a href="https://github.com/w1ll1am23">@w1ll1am23</a> - <a href="https://github.com/home-assistant/core/pull/44427">#44427</a>) (<a href="/integrations/econet/">econet docs</a>)</p>
  </p>
</details>
<details>
  <summary><b>Intents</b></summary>
  <p>
<p>The method for matching entities in intents has changed slightly to prefer
shorter matches before ordering.</p>
<p>This means some intents and/or entity names may need to be adjusted to maintain
consistency in some edge cases. For most uses, this should only improve
the quality of matches.</p>
<p>(<a href="https://github.com/CtrlZvi">@CtrlZvi</a> - <a href="https://github.com/home-assistant/core/pull/43672">#43672</a>)</p>
  </p>
</details>

    <title type="html"><![CDATA[Security Disclosure 2: vulnerabilities in custom integrations HACS, Font Awesome and others]]></title>
    <link href="https://www.home-assistant.io/blog/2021/01/23/security-disclosure2/"/>
    <updated>2021-01-23T00:00:00+00:00</updated>
    <id>https://www.home-assistant.io/blog/2021/01/23/security-disclosure2</id>
    <content type="html"><![CDATA[<p><img src="/images/blog/2021-01-security-disclosure2/social.png" alt="Attention please read" /></p>
<p>This blog looks pretty much the same <a href="/blog/2021/01/22/security-disclosure/">as the security disclosure of yesterday</a>. However, <strong>it is a new disclosure</strong>, affecting a similar issue. We want to make sure the information is complete.</p>
<p><em>This is a disclosure about security vulnerabilities found in <b>3rd party custom integrations</b>. Custom integrations are not created and/or maintained by Home Assistant. Users install them at their own risk. We want to inform you about these because the found vulnerabilities impact the security of your Home Assistant instance.</em></p>
<p><em>If you do not use custom integrations, your Home Assistant is not vulnerable. If you do use custom integrations, your instance might be vulnerable if you use one of the impacted integrations.</em></p>
<p>TL;DR:</p>
<ul>
<li>Multiple custom integrations were found that allowed an attacker to steal any file without logging in. Previously implemented fixes were not sufficient.</li>
<li>Upgrade Home Assistant as soon as possible. Home Assistant Core <strong>2021.1.5</strong> added mitigation to prevent the issue from happening.</li>
<li>Upgrade the custom integrations to a fixed version or remove them from your installation.</li>
<li>If you have used any of the custom integrations with a known vulnerability, we recommend that you update your credentials.</li>
</ul>
<p>On the morning of Saturday, January 23 2021, the Home Assistant project was informed by <a href="https://twitter.com/NateBrady23">security researcher Nathan Brady</a> about a security vulnerability. It provided more insight on the implementation of the fixes done for the <a href="/blog/2021/01/22/security-disclosure/">previous security vulnerability</a>. We learned that not all custom integrations that implement security patches are sufficient to deflect the problem.</p>
<p>We verified all fixes made to custom integrations that were found to be vulnerable in the previous security disclosure. The conclusion is that some custom integrations are still vulnerable to a directory traversal attack while not being authenticated with Home Assistant. It allows an attacker to access any file without having to log in. This access includes any credentials that you might have stored to allow Home Assistant to access other services.</p>
<p>We have responsibly disclosed these issues to the authors of those custom integrations and worked with them on fixing their integrations.</p>
<p>The following have been found:</p>
<ul>
<li><a href="https://github.com/hacs/integration">Home Assistant Community Store</a> (HACS) ‚Äì fixed in <strong>1.10.1</strong></li>

<p><em>This is a disclosure about security vulnerabilities found in <b>3rd party custom integrations</b>. Custom integrations are not created and/or maintained by Home Assistant. Users install them at their own risk. We want to inform you about these because the found vulnerabilities impact the security of your Home Assistant instance.</em></p>
<p><em>If you do not use custom integrations, your Home Assistant is not vulnerable. If you do use custom integrations, your instance might be vulnerable if you use one of the impacted integrations.</em></p>
<p>TL;DR:</p>
<ul>
<li>Multiple custom integrations were found that allowed an attacker to steal any file without logging in.</li>
<li>Upgrade Home Assistant as soon as possible. Home Assistant Core 2021.1.3 added extra protections that stops attackers from reaching the vulnerable code in custom integrations.</li>
<li>Upgrade the custom integrations to a fixed version or remove them from your installation.</li>
<li>If you have used any of the custom integrations with a known vulnerability, we recommend that you update your credentials.</li>
</ul>
<p>On the morning of Thursday, January 14 2021, the custom integration Home Assistant Community Store (HACS) project was informed by <a href="https://www.linkedin.com/in/oriel-goel/">security researcher Oriel Goel</a> about a security vulnerability. It was vulnerable to a directory traversal attack via an unauthenticated webview, allowing an attacker to access any file that is accessible by the Home Assistant process. This access includes any credential that you might have stored to allow Home Assistant to access other services.</p>
<p>We started to research what other custom integrations could be impacted and found several more. We have responsibly disclosed these issues to the authors of those custom integrations and worked with them on fixing their integrations.</p>
<p>The following have been found:</p>
<ul>
<li><a href="https://github.com/hacs/integration">Home Assistant Community Store</a> (HACS) ‚Äì fixed in 1.10.0</li>

<li><a href="https://github.com/thomasloven/hass-fontawesome">Font Awesome</a> ‚Äì fixed in 1.3.0</li>
<li><a href="https://github.com/akasma74/Hass-Custom-Alarm">BWAlarm (ak74 edition)</a> ‚Äì fixed in 1.12.8</li>
<li><a href="https://github.com/vigonotion/hass-simpleicons">Simple Icons</a> ‚Äì fixed in 1.10.0</li>
<li><a href="https://github.com/custom-components/custom_updater/">Custom Updater</a> (deprecated) ‚Äì fixed in latest commit</li>
</ul>
<p>We haven‚Äôt been able to get in touch with the authors of the following integration. You should remove this custom integration as soon as possible:</p>
<ul>
<li><a href="https://github.com/Armaell/home-assistant-custom-icons-loader">Custom icons</a> ‚Äì not fixed</li>
</ul>
<p>The following integration was discovered to be vulnerable to a variant of the above security vulnerability. It allows for a directory traversal attack but requires the attacker to be authenticated. We have been unable to reach the author:</p>
<ul>
<li><a href="https://github.com/yunsean/hass-album/">Hass-album</a> ‚Äì not fixed</li>
</ul>
<p>If you have used any of these custom integrations, we recommend that you update your credentials.</p>
<p>Besides working with the custom integration authors, the following actions have been taken to help protect users:</p>
<ul>
<li>Home Assistant released Home Assistant Core 2021.1.3 with extra protection to stop directory traversal attacks before reaching the vulnerable code. This prevents the abuse of all found vulnerabilities.</li>
<li>Home Assistant published a <a href="https://www.home-assistant.io/blog/2021/01/14/security-bulletin/">security bulletin</a> strongly urging people to upgrade their Home Assistant instance. This bulletin has been shared widely and linked from banners on the Home Assistant website and forums.</li>
<li>The Home Assistant Supervisor will notify the user when a possible insecure installation is found that uses custom integrations.</li>
<li>The Home Assistant Companion apps for Android and iOS have been updated to notify the user if their Home Assistant instance is potentially insecure.</li>
<li>Nabu Casa emailed the security bulletin to all Home Assistant Cloud subscribers and users on trial.</li>
<li>Nabu Casa activated their feature to <a href="https://www.nabucasa.com/config/remote/#insecure-versions">limit remote access</a> via Home Assistant Cloud and block instances that run an insecure version of Home Assistant.</li>
</ul>
<p>Look. It sucks that this happened. The custom integrations we have listed are all open source, maintained by volunteers in their spare time. They often work alone on this and that‚Äôs why it‚Äôs more likely for a bug to go undetected. But more eyes doesn‚Äôt guarantee bug-free software either. From time to time, such things will happen to every piece of software.</p>
<p>I want to emphasize that it‚Äôs not allowed to personally harass/attack/insult the developers of these custom integrations. That would be a violation of our Code of Conduct and we will enforce this.</p>
<p>As Home Assistant, we could have done more to prepare for this scenario. We are currently exploring adding new opt-in features for users to be notified and allow Home Assistant to take action preemptively to patch vulnerabilities.</p>
<p>Paulus</p>

<h2>Breaking Changes</h2>
<p>Below is a listing of the breaking change for this release, per subject or
integration. Click on one of those to read more about the breaking change
for that specific item.</p>
<details>
  <summary><b>M√©t√©o-France</b></summary>
  <p>
<p>The unit of measurement of the UV sensor was missing and has been added. You
may have to adapt your configuration accordingly, if you were depending on the
unit of this sensor in any automations, scripts or 3rd party data loggers.</p>
<p>(<a href="https://github.com/mbo18">@mbo18</a> - <a href="https://github.com/home-assistant/core/pull/43992">#43992</a>) (<a href="/integrations/meteo_france/">meteo_france docs</a>)</p>
  </p>
</details>
<details>
  <summary><b>Glances</b></summary>
  <p>
<p>Previously, the Glances integration was presuming that all sensors were
temperatures. Now, we can distinguish temperatures, fan speeds and battery
charges.</p>
<p>In consequence, those entities have new Entity IDs, so you should update your
configuration in the UI or in your YAML files.</p>
<p>Previously, they were all suffixed by <code>_temp</code>. Now they are suffixed by
<code>_temperature</code>, <code>_fan_speed</code> and <code>_charge</code>.</p>
<p>Examples:</p>
<ul>
<li><code>sensor.glances_core_0_temp =&gt; sensor.glances_core_0_temperature</code></li>
<li><code>sensor.glances_battery_temp =&gt; sensor.glances_battery_charge</code></li>
<li><code>sensor.glances_thinkpad_1_temp =&gt; sensor.glances_thinkpad_1_fan_speed</code></li>
</ul>
<p>(<a href="https://github.com/guix77">@guix77</a> - <a href="https://github.com/home-assistant/core/pull/43500">#43500</a>) (<a href="/integrations/glances/">glances docs</a>)</p>
  </p>
</details>
<details>
  <summary><b>Neato</b></summary>
  <p>
<p>Neato now uses a new authentication method: OAuth2.</p>
<p>If you are an existing Neato user, you will need to update your YAML
configuration, as per the updated <a href="/integrations/neato/">Neato documentation</a>. The
integration will then automatically be migrated.</p>
<p>Vorwerk support is dropped from the neato integration. The parent library still
supports Vorwerk however, the authentication is completely different and
therefore will require a separate integration created.</p>
<p>(<a href="https://github.com/Santobert">@Santobert</a> - <a href="https://github.com/home-assistant/core/pull/44031">#44031</a>) (<a href="/integrations/neato/">neato docs</a>)</p>
  </p>
</details>
<details>
  <summary><b>Tado</b></summary>
  <p>
<p>The Tado Bridge sensor has been removed and replaced by its corresponding device
sensor.</p>
<p>(<a href="https://github.com/Noltari">@Noltari</a> - <a href="https://github.com/home-assistant/core/pull/44475">#44475</a>) (<a href="/integrations/tado/">tado docs</a>)</p>
  </p>
</details>
<details>
  <summary><b>OpenTherm Gateway</b></summary>
  <p>
<p>This release breaks <code>binary_sensor</code> and <code>sensor</code> entities of the <code>opentherm_gw</code>
integration. More specifically, all existing <code>binary_sensor</code> and <code>sensor</code>
entities are recreated with a different <code>entity_id</code>, <code>unique_id</code> and friendly
name.</p>
<p>Some entities are recreated once, others will be recreated twice for different
data sources (boiler and thermostat). All entities that are recreated are
disabled by default.</p>
<p>The old <code>sensor</code> and <code>binary_sensor</code> entities will log a deprecation warning
and change behavior slightly if they are enabled. They will be automatically
removed if they are disabled. Behavioral changes of the old entities may include
a delayed update and a possible change of data source (boiler vs. thermostat)
which may only become apparent when an override is active on the OpenTherm
Gateway.</p>
<p>The <code>climate</code> entity is not affected.</p>
<p>The OpenTherm Gateway sits between the thermostat and the boiler and thus deals
with 2 streams of information. Before this release, the status info we received
contained overlapping information from both streams of information, which led
to issues when the gateway was overriding some values.</p>
<p>As of this release, the status is split up in multiple parts to reflect this.
Each part contains the status of its respective information stream. With this
change, we create sensor entities for each stream where necessary.</p>
<p>(<a href="https://github.com/mvn23">@mvn23</a> - <a href="https://github.com/home-assistant/core/pull/43352">#43352</a>) (<a href="/integrations/opentherm_gw/">opentherm_gw docs</a>)</p>
  </p>
</details>
<details>
  <summary><b>BMW Connected Drive</b></summary>
  <p>
<p>BMW Connected Drive has been migrated for configuration via the UI.</p>
<p>Your old entries from <code>configuration.yaml</code> are automatically imported <strong>once</strong>
for initial setup.</p>
<p>It is recommended to rename the device_tracker of your car(s) in
<code>config/known_devices.yaml</code> (you can e.g., add <code>_old</code> to the entity name)
before updating, otherwise a second device tracker entity (ending with <code>_2</code>)
per car will be created.</p>
<p>After updating, the entity can be removed from known_devices as that will no
longer be used.</p>
<p>(<a href="https://github.com/rikroe">@rikroe</a> - <a href="https://github.com/home-assistant/core/pull/39585">#39585</a>) (<a href="/integrations/bmw_connected_drive/">bmw_connected_drive docs</a>)</p>
  </p>
</details>
